Middleware compatibility note (Pitfall — 11-RESEARCH.md):
better-auth recommends Node.js runtime in middleware (Next.js 15.2+) for full session validation.
For Edge-compatible cookie presence check (no DB query), use `getSessionCookie` from
`better-auth/cookies` — returns true/false based on cookie existence.

Strategy: middleware does an Edge-friendly cookie-presence check only (fast — no Neon round-trip per request).
Per-page layout.tsx and per-API-route handlers do the AUTHORITATIVE `auth.api.getSession()` check
(server-side, full validation against the sessions table). This matches the research's
"belt-and-braces" admin guard pattern (CVE-2025-29927) preserved from Phase 6.

  - account   (id, userId, accountId, providerId, password, createdAt, updatedAt)
  - verification (id, identifier, value, expiresAt, createdAt, updatedAt)

These MUST be added to `src/db/schema.ts` so they participate in the same Drizzle migration system.

      const password = formData.get('password') as string ?? ''

          asResponse: false,        // we want the API to set cookies via headers, not return a Response

    }
