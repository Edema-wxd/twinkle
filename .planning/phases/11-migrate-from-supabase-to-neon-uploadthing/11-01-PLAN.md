<objective>
Replace Supabase PostgreSQL with Neon serverless PostgreSQL accessed via Drizzle ORM. This wave installs Drizzle + the Neon HTTP driver, defines the full schema for all 10 production tables, pushes that schema to Neon, then systematically replaces every `supabase.from('...')` and inline `createClient` data-access call with a Drizzle query. The Supabase auth helpers (`supabase.auth.getUser()`) and storage calls are intentionally LEFT IN PLACE — they are removed in Plans 02 and 03 respectively.

Purpose: Establish a typed, schema-driven query layer that becomes the single source of truth for the database. Eliminates the hand-maintained `src/types/supabase.ts` (replaced by `$inferSelect`). Removes the Supabase JS client from data-fetch paths so that Plan 02 (auth) and Plan 03 (storage) can execute against a stable foundation.

Output: `src/db/schema.ts`, `src/db/index.ts`, `drizzle.config.ts`, all 10 tables provisioned in Neon, and ~38 application files migrated from `supabase.from()` to `db.select()/insert()/update()/delete()`. Auth call sites are unchanged in this plan.
</objective>

Existing query patterns that MUST keep working after migration:
  - .eq('is_active', true)              → eq(products.isActive, true)
  - .eq('slug', slug).maybeSingle()     → .where(eq(...)).limit(1)  →  result[0] ?? undefined

  - .order('created_at', { ascending: false })  → orderBy(desc(table.createdAt))
