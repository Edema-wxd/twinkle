    Update `src/app/api/webhooks/paystack/route.ts` so admin email notification is attempted for a `charge.success` event in an idempotent way:

    1) Change the current idempotency guard:
       - Today it returns early when an order already exists.
       - New behavior: if the order exists, still proceed to the notification path (do NOT re-insert the order or items).
       - Concretely: fetch the existing order row ID (and minimal fields needed) by `paystack_reference` and store `orderId`.

    2) After ensuring `orderId` (either newly inserted or existing), perform notification logic:
       - Call `const to = await getAdminNotificationEmail()`
       - Call `const notif = await ensureOrderNotification({ orderId, channel: 'email' })`
       - If `notif.status === 'sent'`: return (do nothing)
       - If `notif.attempts >= 3`: return (do nothing; prevents infinite retries/spam)
       - Otherwise attempt to send:
         - Build the concrete `order` input for `sendAdminOrderEmail` using:
           - `reference`: Paystack reference
           - `customerName`, `customerEmail`, `customerPhone`, `deliveryState`, `totalKobo` (from `orders.total`)
           - `itemCount`: use `cart_items.length` from metadata when available; if metadata missing on duplicates, fallback to `0` (do NOT throw)
         - `await sendAdminOrderEmail({ to, order: ... })`
         - On success: `await markOrderNotificationSent({ id: notif.id })`

       - On failure (any thrown error from send/env/provider):
         - `const { attempts } = await markOrderNotificationFailed({ id: notif.id, error: String(err) })`
         - If `attempts < 3`, rethrow the error so the outer handler returns 500 and Paystack retries.
         - If `attempts >= 3`, swallow (do not throw) so the webhook can return 200 and stop retries.

    3) Security constraints:
       - Do not log full webhook bodies or provider secrets.
       - It is OK to log: reference + notification id + attempt count + error message (truncated).
  </action>
